# UAAD 贡献指南 (CONTRIBUTING)

本项目采用 **多人并行 + AI 辅助** 的协作开发模式。本文档说明协作过程中必须遵守的规则。

---

## 1. 先文档，后代码

- 开工前先写技术方案/接口定义/页面设计思路，组内对齐后再动手写代码
- 避免"代码写了不知道干嘛"和"前后端接口对不上"
- 方案完成后及时更新对应文档，不要只写代码不更新文档

## 2. API 契约是团队的边界协议

- `docs/SYSTEM_DESIGN.md` §4 是唯一的 API 真理源
- 前端按契约写类型和 Mock，后端按契约实现接口
- **API 变更必须先更新 SYSTEM_DESIGN.md，再通知对应组**
- PR Review 时对照 API 契约检查响应体格式、状态码、分页字段

## 3. 文件隔离，不碰别人的代码

- 每组有自己的 domain/repository/service/handler 文件，不跨组修改
- `main.go` 由 Tech Lead 统一合并路由，其他人通过 `*_setup.go` 注册函数
- `App.tsx` 由前端负责人统一注册路由，其他人只提交页面/组件文件
- 遇到文件冲突由 PR 发起方负责解决

## 4. 分支策略

- 所有开发在 `feature/*` 分支进行，不直接向 `main` 提交
- 合并到 `dev` 需要 PR + 至少 1 人 Review
- 每日同步 `dev` 到本地：`git pull origin dev`
- feature 分支不超过 3 天，频繁合并减少冲突累积

## 5. 不超 SRS 范围开发

- 所有功能和接口以 `docs/SRS.md` 和 `docs/SYSTEM_DESIGN.md` 为准
- AI 生成代码时严格遵循文档定义，不擅自添加文档中没有的功能
- 有额外想法先讨论，确认后再更新文档再实现

## 6. 各组独立推进，互不等待

- 前端用 MSW Mock 开发，不等后端接口完成
- 后端各组之间业务模块互不依赖，同步推进
- 联调阶段再切真实后端，对照 API 契约对齐

## 7. 测试与质量

- 后端核心链路必须写测试（报名并发、幂等、权限校验等关键路径）
- 前端页面需适配中英文，处理加载和异常状态
- PR 必须包含变更描述和验证方式（curl 截图或页面截图）

## 8. 沟通节奏

- 每天站立会：完成、计划、阻塞
- API 变更随时通知，不要默默改了接口让别人调不通
- Sprint 结束后做集成测试，全链路跑通再进入下一阶段

## 9. 文档维护

- **非必要不新建文档**，优先更新已有文档
- `docs/SPRINT1.md` — 当前分工入口
- `docs/SYSTEM_DESIGN.md` — API 契约与系统架构
- `docs/COLLABORATION.md` — 团队冲突预防与流程细节
- `docs/SRS.md` — 需求规格
- `docs/DOCS_PROMPTS_GOVERNANCE.md` — 文档与提示词治理入口
- `.agents/` 是团队 AI 工作流真理源；`.github/*` 与其他工具目录默认作为适配层或分发副本维护
- 修改 Prompt / Workflow 时，先改真理源，再修入口引用，最后再同步镜像目录
