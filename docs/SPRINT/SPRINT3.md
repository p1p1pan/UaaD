# UAAD Sprint 3 工作划分大纲

> **核心目标：补齐 Sprint 2 遗留闭环、强制完成完整闭环测试与大规模压力测试、固化验证基线、提升系统可观测性与交付可信度**
>
> **前置说明：** 当前仓库已具备活动、报名、订单、通知、推荐、商户端与基础中间件能力，但从真实验收口径看，仍存在若干关键缺口：如订单过期通知未接线、前端行为埋点未落地、Kafka Worker 与监控能力未完全达到预期、部分联调与文档证据链仍需收口。Sprint 3 的目标不是继续横向铺新功能，而是把系统推进到“真正闭环、可验证、可演示、可答辩”的状态。
>
> 最后更新：2026-04-18

---

## 协同与提交流程规范

> **⚠️ 本 Sprint 的原则：先收口，再扩展。**
> 1. 任何涉及 API、状态机、通知触发点、埋点格式的变更，必须先更新 `docs/SYSTEM_DESIGN.md` 对应章节。
> 1.1 新增业务需求与跨模块变更（非文案修补）进入开发前，需建立或更新对应的 OpenSpec change。
> 1.2 **合规审查**：代码审查（CR）要求 `openspec/changes/.../tasks.md` 中的对应任务已勾选，且执行 `openspec validate` 无报错。
> 2. 本 Sprint 优先处理链路闭环、完善验收证据与可观测性，暂缓低优先级功能。
> 3. 所有组需提供可复现验证方式，不接受“代码看起来对了但没有证据”的提交。
> 4. 若发现 Sprint 2 文档与当前代码不一致，应在本 Sprint 内同步修正，而不是继续让实现与文档分叉。
> 5. **完整闭环测试与大规模压力测试属于 Sprint 3 的强制交付项，不是可选优化项。**

## 二、Sprint 3 工作背景与收口原则

当前项目已具备以下基础：

- 后端：MySQL + Redis + Kafka + Prometheus 基础指标、活动/报名/订单/通知/推荐等核心模块。
- 前端：公开首页、活动广场、活动详情、报名状态、订单、通知、商户控制台等页面已建立。
- 联调：Axios 鉴权、受保护路由、商户权限流、JMeter 脚本与基础运行文档已存在。

但从当前代码审查结果看，仍有下列遗留风险：

- **订单过期通知未接线**：订单关闭后库存会回补，但用户并不会收到 `ORDER_EXPIRE` 通知。
- **前端行为埋点缺失**：后端已有 `/behaviors` 与 `/behaviors/batch`，但前端关键页面尚未真实上报 `VIEW` / `CLICK` / `SEARCH`。
- **异步链路观测不足**：Prometheus 已记录 HTTP 指标，但 Kafka Worker 处理质量、消费积压与失败路径缺少明确观测。
- **文档与验收证据需固化**：当前已有运行说明和压力测试脚本，但 Sprint 3 需要把它们整理成可直接验收的证据链。
- **完整闭环测试尚未被提升为硬性门槛**：当前存在联调检查与局部压测材料，但还缺少“从商户建活动到用户报名、订单流转、通知触达”的统一闭环验收。
- **压力测试规模仍偏保守**：已有 1000 线程级压测基础，但 Sprint 3 需要进一步完成更大规模压测并给出结果分析。

本 Sprint 的原则是：

- **优先补闭环，不优先加新页面**
- **优先出验收证据，不优先写“看起来完整”的文档**
- **优先消除真实断点，不做大而空的架构重写**
- **闭环测试与压力测试不过关，则 Sprint 3 不算完成**

## 三、第一组：Backend 闭环收口与可靠性补齐

**核心模块：** `service/order_service.go`、`service/notification_service.go`、`worker/enrollment_worker.go`、`service/enrollment_service.go`、相关 repository / tests。

**关键改造任务：**
1. **订单过期通知接线：**
   - 在订单过期扫描逻辑中正式注入 `NotificationService`。
   - 对 `ScanExpired()` 中每一笔成功关闭的订单，写入 `ORDER_EXPIRE` 通知。
   - 补齐活动标题获取逻辑，确保通知文案不出现空标题或 `unknown` 占位。
2. **通知链路完整性复核：**
   - 校验报名成功、报名失败、订单过期三类通知的 `type`、`related_id` 与前端跳转约定是否一致。
   - 确保通知创建失败不拖垮主链路，但必须有稳定日志输出。
3. **异步报名链路吞吐量优化：**
   - 评估 Kafka 消费端吞吐能力，避免因单条处理造成队列积压及前端 `QUEUING` 超时。根据压测结果，必要时引入批量消费（Batch Consume）策略。
   - 校验 Kafka 入队、MySQL 事务及 Redis 回补的强一致性。
   - 补充对唯一键冲突、库存回补失败、消息体反序列化异常等边界测试。
4. **状态变更原子性与一致性修复：**
   - **状态机控制**：`UpdateStatus` 需引入乐观锁机制（如 `WHERE id = ? AND status = 'PENDING'` 配合 `RowsAffected` 校验），解决 `Pay` 与 `ScanExpired` 并发时的 TOCTOU 问题，保障压测下的库存安全性。
   - 修复 Worker 消费成功后未能同步更新 MySQL 活动表 `enroll_count` 的缺陷。
   - 对齐报名状态（`QUEUING/SUCCESS/FAILED`）与订单状态（`PENDING/PAID/CLOSED`）的业务逻辑流转。
   - 验证订单关闭后的 Redis 与 DB 库存补偿幂等性，防止重复回补。
5. **后端测试补强：**
   - 新增订单过期通知测试。
   - 新增或补强 Worker 冲正与通知触发测试。
   - 为关键业务状态流补齐黑盒或集成级测试。
6. **闭环测试后端支撑：**
   - 为完整闭环测试提供稳定 seed 数据、商户账号、用户账号与可复用脚本。
   - 确保闭环链路中的关键事件可从日志、数据库与接口结果中回查。
7. **补齐排队与未支付状态的自主取消机制：**
   - 该能力必须先对齐或继续推进对应 OpenSpec change，并同步 `docs/SYSTEM_DESIGN.md` 中的 API 契约、状态流转与错误码定义。
   - 提供 `POST /enrollments/:id/cancel` 接口，支持用户主动取消处于 `QUEUING` 状态的报名，以及已生成 `PENDING` 订单但尚未支付的报名。
   - 用于验证闭环场景下的即时库存补偿逻辑，确保异步队列在并发与强行终止边界的健壮性（已支付订单的资金退回逻辑本期暂不涉及）。
8. **热度计算与状态流转口径收口：**
   - 保持与 `docs/SYSTEM_DESIGN.md` 一致：热度评分采用“定时全量重算 + 事件触发增量更新”的模型，不在本 Sprint 内额外引入与设计文档冲突的第二套调度架构。
   - 若要实现活动逾期转 `OFFLINE`，应明确它属于活动状态维护任务，与推荐热度计算分开描述、分开验收。
   - 跑通从“前端行为数据采集 -> 行为写库 -> 热度评分更新 -> 推荐排序变化可观测”全链路，确保前端行为采集数据能够在展示维度生效。

**验收标准（DoD）：**
- 对于一笔超时未支付订单，系统可稳定完成：`订单关闭 -> 库存回补 -> 通知写入`。
- 对于报名成功与事务失败两类路径，通知表可分别产生 `ENROLL_SUCCESS` / `ENROLL_FAIL` 记录。
- 压测或并发测试后，Redis 与 DB 不出现负库存，也不出现重复成功报名。
- 对于主动取消路径，系统可稳定完成：`取消请求 -> 状态转移 -> 库存回补 -> 幂等防重`，且不会与 `Pay()` / `ScanExpired()` 产生重复回补。
- 前端行为埋点写入后，必须能在至少一条推荐链路或热度结果上观测到生效证据，不接受“只上报、不验证结果”的交付。
- 所有后端闭环关键节点均可被测试脚本或验收清单稳定复现，不依赖人工猜测状态。

## 四、第二组：Observability、压测与验收基线建设

**核心模块：** `middleware/metrics.go`、`tests/stress_test.go`、`tests/jmeter/*`、`docs/STRESS_TEST.md`、`docs/RUN_GUIDE.md`、`docs/Prometheus_And_Grafna.md`。

**关键工作任务：**
1. **Prometheus 指标增强：**
   - 保留现有 HTTP 请求总量与时延指标。
   - 为异步链路新增最小必要指标，例如：
     - Worker 消费成功数
     - Worker 消费失败数
     - Worker 单次处理耗时
     - 可选：当前队列处理深度或消费滞后近似值
2. **Grafana 可视化收口：**
   - 形成一套明确可复用的 Dashboard 指标面板建议。
   - 至少能观察：
     - HTTP 时延
     - 请求吞吐
     - Worker 成功/失败
     - 抢票或报名压力测试期间的服务波动
3. **压测基线固化：**
   - 基于现有 JMeter 脚本，跑出统一口径的基线结果。
   - 明确压测主口径：核心压测接口为 `POST /api/v1/enrollments`，热点场景默认针对单个已发布且已完成库存预热的活动进行。
   - 明确业务成功判定：
     - `HTTP 202 + code=1201` 计为“成功进入排队”
     - `HTTP 200/410 + code=1101` 计为“业务性售罄拒绝”
     - 5xx / 超时 / 非预期响应体计为失败
   - 明确压测前置条件：鉴权、种子账号、活动预热、Redis/Kafka/MySQL/Prometheus/Grafana 均已正常启动。
   - 固化输出以下结果项：
     - 成功率
     - P95 / P99
     - 5xx 是否出现
     - 最终库存是否严格不为负
     - 是否存在重复成功报名
4. **完整闭环测试验收清单建设：**
   - 制定 Sprint 3 专用闭环验收清单，至少覆盖以下全链路：
     - 商户登录
     - 商户创建并发布活动
     - C 端登录
     - 用户浏览首页 / 活动广场 / 活动详情
     - 用户发起报名进入 `QUEUING`
     - Worker 落盘为 `SUCCESS` 并生成 `PENDING` 订单
     - 用户完成支付，或订单超时关闭
     - 通知页与通知铃铛出现对应业务通知
     - 行为埋点写入成功
   - 对每一环节给出验证方式、预期结果与证据位置。
5. **大规模压力测试实施（Sprint 3 强制项）：**
   - 在既有 1000 线程基础上，执行一轮更大规模压测，目标规模建议为：
     - **基础目标：3000 并发请求**
     - **冲刺目标：5000 并发请求**
   - 结果必须绑定同一套压测模型，不允许不同人使用不同接口、不同活动基线或不同成功判定后直接比较“3000/5000 并发结果”。
   - 若受本机硬件或环境限制无法一次性达到冲刺目标，必须记录瓶颈原因、降级方案与实际达到的规模。
   - 压测结果必须形成报告，不接受仅口头描述“跑过了”。
6. **压测报告输出：**
   - 报告至少包含：
     - 测试日期
     - 环境配置
     - 线程数 / Ramp-up / 持续时间
     - 成功率
     - P95 / P99
     - 5xx / 超时数量
     - 最终库存一致性结果
     - 关键监控截图
7. **验收文档整理：**
   - 更新 `docs/STRESS_TEST.md` 与 JMeter 验收清单，使其与当前代码完全对应。
   - 明确哪些能力已经具备，哪些监控仍为基础版。
8. **运行链路清晰化：**
   - 更新 `docs/RUN_GUIDE.md`，确保新同学可按文档拉起 MySQL / Redis / Kafka / Prometheus / Grafana 与前后端服务。
   - 明确说明开发态、联调态、演示态各自的配置要求。

**验收标准（DoD）：**
- 可以通过既有脚本与文档，稳定复现一轮**完整闭环测试**与至少一轮**高并发报名压测**。
- 至少一套 Prometheus / Grafana 面板能展示服务时延与异步处理状态。
- 压测结论可被文档、日志与监控三方交叉验证。
- 压测报告必须明确记录所压接口、活动基线、线程数、Ramp-up、持续时间、成功判定和失败判定。
- Sprint 3 必须产出一轮**大规模压力测试报告**，规模不低于 3000 并发；若未达到 5000 并发，应明确说明限制条件。

## 五、第三组：前端联调收口、埋点回流与体验打磨

**核心模块：** `frontend/src/api/endpoints/`、`pages/Home.tsx`、`pages/PublicActivities.tsx`、`pages/ActivityDetail.tsx`、`pages/Notifications.tsx`、`components/public/PublicHeader.tsx`、相关 hooks / utils / i18n。

**关键优化任务：**
1. **前端行为埋点落地：**
   - 新增 `behaviors.ts` 请求封装并接入导出。
   - 在关键交互页接入最小行为采集闭环：
     - 首页 / 推荐位点击 → `CLICK`
     - 活动详情进入 → `VIEW`
     - 搜索提交 → `SEARCH`
   - 考虑引入 `batch` 缓冲上报或预留扩展点。
   - **异步无阻塞上报**：埋点请求应设置合理的短超时机制，并采用 Fire-and-forget 策略，确保埋点失败时不影响核心业务流。
   - 埋点数据结构需严格对齐 `docs/SYSTEM_DESIGN.md` 中的 `/behaviors` 与 `/behaviors/batch` 规范。
2. **通知体验统一：**
   - 统一顶部通知铃铛、通知页、未读数与已读状态同步逻辑。
   - 优化预览面板、详情弹层、通知页的交互一致性。
   - 视情况评估未读数是否直接走 `/notifications/unread-count`，降低整表拉取成本。
3. **页面边界态与防抖控制：**
   - 统一订单页、活动详情页、通知页、商户端等模块的 Skeleton / Empty / Error / Retry UI 表现。
   - 消除无说明的空白报错页及缺失重试路径的异常状态。
   - **异步操作控制**：对包含 `QUEUING` 在内的异步行为，增加防抖与 Loading 锁，并在接口回调后主动轮询或拉取最新状态，以保证前后端数据展示一致。
4. **移动端与断点复验：**
   - 重点检查 `<768px` 下首页、活动广场、活动详情、订单页、商户活动列表的溢出与换行。
   - 修正主要布局错乱、按钮遮挡、卡片信息截断等问题。
5. **Mock 策略规范化：**
   - 明确 `VITE_USE_MOCK` 的适用场景。
   - 联调与验收阶段默认关闭 Mock；本地开发允许按需启用。
   - 将此策略写入运行文档，避免因环境变量误用导致联调结果失真。
6. **前端闭环验收配合：**
   - 配合完整闭环测试，确保前端端到端链路可稳定复现，不出现“刷新后状态丢失”“跳转断链”“通知列表未同步”等问题。
   - 为闭环测试提供页面侧验证入口与必要提示文案，降低人工验收成本。

**验收标准（DoD）：**
- 前端关键页面能够向后端真实写入行为数据，且不阻塞主交互。
- 通知铃铛、通知页、未读状态与真实后端数据一致。
- 在桌面端与移动端下，主要页面不存在明显错位、溢出或不可操作问题。
- 至少展示一条“前端埋点触发 -> 后端入库 -> 推荐或热度结果变化”的端到端证据。
- 前端可完整支撑 Sprint 3 闭环测试，不依赖临时改代码或手工跳过页面步骤。

## 六、集成流转与依赖关系

```text
【 第一组: 后端闭环收口 】
    ↳ 先补齐订单过期通知、异步链路状态与后端测试
           ↓
【 第二组: 可观测性、闭环测试与压测基线 】
    ↳ 在稳定业务链路基础上固化闭环验收、指标、压测脚本与大规模压力测试证据
           ↓
【 第三组: 前端联调与体验收口 】
    ↳ 将行为埋点、通知体验、边界态与移动端体验全部接到真实后端能力
```

**并行说明：**

- 第一组与第三组可并行推进，但涉及通知 `type`、行为埋点 payload、错误码映射时必须先对齐契约。
- 若推进自主取消接口，第一组必须先完成 OpenSpec / `SYSTEM_DESIGN.md` 契约更新，第三组再接入真实前端动作。
- 第二组可在第一组补齐关键通知链路后开始固化闭环测试、压测与验收数据。
- 若任一组调整 API 字段或状态枚举，需同步更新 `docs/SYSTEM_DESIGN.md`。
- 大规模压力测试建议安排在 Sprint 3 中后段执行，前置条件是后端闭环已补齐、前端联调链路稳定、监控面板可用。

## 七、Sprint 3 预期里程碑

- 完成从“功能存在”到“业务闭环真实成立”的过渡：用户浏览、报名排队、订单生成、支付、订单过期、通知触达形成稳定链路。
- 将行为埋点正式接入前端真实交互页面，为推荐系统提供真实回流数据源。
- 形成可被复现的**完整闭环测试基线**、**大规模压力测试基线**与监控证据，不再仅凭代码结构判断系统可用性。
- 完成 UI 与错误态收口，使系统达到可演示、可答辩、可对外讲解的稳定程度。
- 至少完成一轮 3000 并发的大规模压力测试，并尽可能冲刺 5000 并发。

## 八、版本协同与交付补充准则

1. 本 Sprint 不鼓励引入大体量新依赖；若确需新增日志、监控或测试库，必须先评估收益与维护成本。
2. 任何对通知触发点、埋点事件名、订单状态流、排队状态流的修改，都应视为高风险变更，必须在 PR 中附带验证说明。
3. 所有关键提交应带可复现的验证步骤，至少包含：
   - 触发方式
   - 期望结果
   - 证据位置（日志、接口响应、页面截图或测试命令）
4. Sprint 3 结束时，应至少形成一套完整的演示脚本，覆盖：
   - 商户创建并发布活动
   - 用户浏览并报名
   - 队列转订单
   - 订单支付或订单超时关闭
   - 通知与推荐相关结果展示
5. Sprint 3 结束时，应额外形成两类强制交付物：
   - **完整闭环测试清单与执行证据**
   - **大规模压力测试报告与监控截图**
