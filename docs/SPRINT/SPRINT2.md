# UAAD Sprint 2 工作划分大纲

> **核心目标：落地高并发架构引擎、移除前端 Mock 构建真实业务闭环、优化组件状态与交互体验**
>
> **前置说明：** Sprint 1 已成功交付 DB 层、JWT 鉴权及全套基础业务模块（CRUD）。Sprint 2 将重点解决高可用架构转型，全面集成中间件以消除高并发隐患，并推进前后端全链路联调，确保端到端交付标准。
>
> 最后更新：2026-04-07

---

## 协同与提交流程规范

> **⚠️ 依赖解决与集成顺序：**
> 1. 第一组（基础设施与中间件）：须优先交付稳定可运行的 `docker-compose.yaml` 及最小环境拓扑，并在团队确认后，其它组方可进行环境同步测试。
> 2. 第二组（前后端联调）：优先移除前端 Mock 服务，确保核心业务链路的数据流转畅通；面对端侧字段或类型映射不对齐的问题，需遵循统一契约并第一时间落实修正。
> 3. 沟通机制：鉴于涉及大量分布式状态的扭转与长连接处理，各组需保证 代码与进度对齐频率。

## 二、运行环境演进说明

开发阶段架构从单体数据库形态（MySQL）向微服务高可用组件拓扑（MySQL + Redis + MQ + 监控件）演进。

- **依赖编排更新**：更新 `docker-compose.yaml`，引入 `Redis 7` 与 `Kafka (基于 Kraft 模式)`，并同步加入简易 `Prometheus` 与 `Grafana` 监控容器。
- **环境隔离要求**：各开发者需维护本机的 `.env` 或 `config.go`，确保各类数据源 URI 可动态配置。
- **Redis 键空间规范**：活动核心库存键约束为 `activity:{id}:stock`；防重复报名集合约束为 `activity:{id}:enrolled_set`；全局排队号为 `activity:queue:global:counter`（严格遵循 DDL 设计约定）。

## 三、第一组：基础设施升级与中间件改造

**核心模块：** 编排文件 `docker-compose.yaml`、核心服务 `service/enrollment_service.go`、异步处理器 Worker。

**关键改造任务：**
1. **环境基座搭建：** 输出标准化的 `docker-compose.yaml`，确保具备幂等拉起能力，供各团队同步。
2. **基于 Lua 的防超卖库存引擎：**
   - 移除现有 MySQL 乐观锁机制。
   - 严格遵循引擎设计实现 `EVAL` Lua 脚本：在一个原子操作内完成 1. 幂等校验（`SISMEMBER enrolled_set` 返回 -2）；2. 库存校验（`<= 0` 返回 -1）；3. `DECRBY`/`SADD`双写原子扣减，并自增返回 `global:queue` 排队号。
3. **基于 Kafka 的异步排队系统：**
   - 优化 HTTP 请求链路：鉴权与频控放行后，封装 Message 投递至 Kafka 的 `enrollment_requests` Topic。
   - 接口响应重构：变更为异步返回 `HTTP 202 Accepted`，携带 Lua 产生的 Queue Position 等排队信息。
4. **订单持久化核心 Worker：**
   - 实现常驻消费组（Consumer）按照 Batch Size 批量消费，执行 MySQL 落盘事务（将报名态更新为 `SUCCESS` 并创建 `PENDING` 订单）。
   - 务必实现冲突兜底冲正：遇唯一键冲突等事务失败时，立即向 Redis 发送回补指令（`INCR stock` + `SREM enrolled_set`）。
   - 时效扫描 Worker：按需提供 5 分钟间隔定时扫描过期订单能力，以用于关闭闭环订单释放 Redis 资源。
5. **缓存击穿防护（预热机制）：** 开发活动发布的预热逻辑接口（Warm-up），支持将预设活动库存量提前同步至 Redis 缓存节点。
6. **简易性能监控基底建设（Observability）：** 利用加入系统的 Prometheus 与 Grafana，为 Go 服务配置基础的指标暴露点，重点能够观测到系统 HTTP 时延及 Kafka 的 Consumer Lag。

**验收标准（DoD）：**
- 依据并发性能测试基准（例如：1000 并发执行库存扣量），执行完毕后验证 DB 与 Redis 剩余可用库存数值准确无误为 0，绝对杜绝负库存异常。

## 四、第二组：前后端闭环联调

**核心模块：** 前端 `src/api` 配置、响应拦截处理及数据注入层调度。

**关键联调任务：**
1. **Mock 服务下线：** 移除本地 MSW 或其他拦截器依赖配置，将 XHR 请求强制转发至本地网关（例如 `localhost:8080/api/v1`）。
2. **视图层真实数据接入：**
   - **首页广场 (`Activities.tsx`)**：对接后端 `/recommendations` 与 `/activities` 接口，替换硬编码数据结构。
   - **详情系统 (`ActivityDetail.tsx`)**：确保视图渲染严格依赖商户后端投递的规范数据。
   - **商户控制台 (`MerchantDashboard.tsx`)**：打通商户权限流与鉴权模块，实现活动的增量更新及流转。
3. **反序列化与字段清洗：** 针对 `Int64` 精度溢出、时间戳序列化等跨语言 (Go 至 TypeScript) 通用问题，于 API 拦截层或组件层建立清洗统一机制。
4. **Token 生命周期验证：** 核查业务页面于 `401 Unauthorized` 时的拦截逃逸保护、状态清理，及登陆回调动作是否健壮。
5. **核心链路并发压测实施（Load Testing）：** 组内需提供简易的 JMeter 测试脚本，借此向底层抢票接口打入（如 1000 个瞬时线程）高频并发，主动诱发系统瓶颈。

**验收标准（DoD）：**
- 基于 Google DevTools 测试：实现全端全流程 Chrome Network 中无异常拒绝请求，突变操作均正确携带认证凭证。
- 压测与监控反馈：执行高压测试后，观测 Grafana 大盘曲线变化平滑，且排查验证库存数值严格对齐并不为负数。

## 五、第三组：前端组件深度优化与缺失业务补全

**核心模块：** 异步交互组件（轮询与通知）、前端数据采集机制、兜底结构设计、表单完整性边界管控。

**关键优化任务：**
1. **C 端订单链路补全与渲染：**
   - 新建 `Orders.tsx` 及 `OrderDetail.tsx` 页面并配置子路由，填补前端订单流转层的结构空白。
   - 映射后端订单流转状态树，完成“待支付”、“排队处理中”、“已取消”、“已闭单”等交易节点的界面渲染与操作闭环。
2. **状态轮询控制与异步站内信展示：**
   - **排队轮询隔离**：配合后端的 Lua 排队特性，构建防抖拦截拦截及加载层组件。开发限频短轮询机制（频率约束：3~5 秒调用 `GET /api/v1/enrollments/:id/status`），取得 `SUCCESS` 后跳转收银支付层。
   - **全局异步消息拉取**：对接后端 `/notifications` 接口，于系统全局视图层渲染消息接收器（如导航栏右侧），及时将诸如死锁冲正导致的被动 `ENROLL_FAIL` 或过期通知传达给终端实例。
3. **前端异步交互行为数据采集（Telemetry）：**
   - 基于后端个性化混合推荐算法对数据的依赖，于关键交互页面（广场组件、详情页）建立非阻塞的端侧用户行为抓取模块（捕获如 VIEW、CLICK 等状态）。
   - 采纳缓冲池机制与防抖策略执行延迟回存请求（如每 60 秒或攒成批次批量发送日志），为数据分析算法源源不断提供可靠的前置特征输入。
4. **边界异常页及应用占位组件：**
   - 对未完善的 `Profile.tsx` 与 `Settings.tsx` 完成基础响应式 DOM 搭建。
   - 对空数组、空对象或 HTTP 异常边界状态引入空状态机处理（Empty State 组件库），屏蔽系统白板现象。
5. **B 端业务强校验表单：**
   - 强化商户提交表单 `MerchantForm` 下的时间序列校验策略。强制前置核验阻断逻辑：确保“报名开始时间 < 报名结束时间 < 活动起始时间”不发生倒反违例。
6. **视区断点（Breakpoints）结构复验：** 验证在极致窄屏及标准移动端断点尺寸（< 768px）下产生的网格换行、多态折叠溢出及边距错乱问题。

## 六、集成流转与依赖关系

```text
【 第一组: 架构与并发基建 】 
    ↳ 交付包含 Kafka、Redis 的拓扑，及提供异步排队控制链路 (QUEUING 状态机)
           ↓
【 第二组: 数据管线联调 】
    ↳ 拆除前端 Mock 网络，桥接第一组交付之接口，校验通讯数据架构的健壮性
           ↓
【 第三组: 业务优化 】
    ↳ 基于第二组确认稳定的数据底座，针对异步长耗时动作补充轮询器、遮罩及各类容错校验机制
```

## 七、Sprint 2 预期里程碑

- 消除对于单体低吞吐状态机的系统瓶颈，交付具备应对多路复用与高可用需求、支持强力高并发队列分发与缓存秒杀的分布式后端基石。
- 实现前端从展示层到链路层的真实对接闭环：全覆盖达成 C 端分发浏览 -> 无阻塞排队下单 -> 后台确认流转；并确保功能层没有业务断层与代码死角。
- 完成用户界面体验升级，操作流与系统阻断状态应具备标准现代化商业系统的视觉传达与工程高可用。

## 八、版本协同防冲突补充准则

1. `frontend/package.json` 的新增依赖（如组件库或校验库）引入，需经技术评审评估，严控框架冗余与代码污染。
2. 执行特性分支往集成分支推拉代码时，应当遵守原子性策略，并使用规范化的强语义化 Commit，以确保溯源审计链路清晰。
3. 若需针对核心鉴权钩子和共享的 Axios 拦截器修改配置或调整参数，执行人应与其它用以依赖的组员提前做好沟通。
