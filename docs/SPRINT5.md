## 二、测试范围

安全测试覆盖当前已进入验收阶段的核心链路：

1. 认证与会话安全：登录、JWT、过期 token、错误 token、401 处理。
2. 权限控制：商户接口、用户接口、受保护页面与后端受保护 API。
3. 越权访问：订单、报名、通知等用户私有资源的水平越权防护。
4. 输入安全：SQL 注入、非法 ID、非法分页、异常 payload。
5. 前端输出安全：活动标题、活动描述、通知内容、用户名等字段的 XSS 风险。
6. 限流与防刷：注册接口限流、重复请求、连击防护。
7. CORS 与生产配置：生产环境跨域白名单、默认密钥、默认账号与 Mock 状态。
8. 敏感信息泄露：响应体、错误信息、日志、环境文件、依赖漏洞扫描结果。
9. 模糊测试：对核心 API 的路径参数、查询参数、JSON body 和鉴权头进行异常输入扰动，验证服务不崩溃、不越权、不返回敏感信息。

## 三、核心任务清单

### 1. 安全测试用例设计

- 梳理后端受保护接口清单，标注是否需要登录、角色要求与资源归属要求。
- 输出安全测试矩阵，至少包含：
  - 未登录访问保护接口应返回 401。
  - 非法 token、过期 token、错误 Authorization 格式应返回 401。
  - CUSTOMER 访问商户接口应返回 403。
  - 用户 A 访问用户 B 的订单、报名、通知应失败。
  - SQL 注入与非法参数不应绕过校验。
  - XSS payload 在前端只能作为普通文本展示。

### 2. 后端自动化安全测试

- 新增或补强后端安全测试文件，例如 `backend/tests/security_authz_test.go`。
- 覆盖以下最小用例：
  - `GET /api/v1/orders` 无 token 返回 401。
  - `POST /api/v1/activities` 使用 CUSTOMER token 返回 403。
  - 用户 A 查看或支付用户 B 的订单失败。
  - 用户 A 查看或取消用户 B 的报名失败。
  - 用户 A 标记用户 B 的通知已读失败。
  - 非法 ID、SQL 注入式 ID、异常分页参数返回 400 或合理业务错误。
- 执行 `go test ./... -count=1`，记录结果。

### 3. 前端与接口手工安全走查

- 使用真实后端和 `VITE_USE_MOCK=false` 执行手工走查。
- 检查以下场景：
  - 未登录访问受保护页面会跳转登录页。
  - 登录过期或 401 后前端清理会话并跳转登录页。
  - 活动标题、活动描述、通知内容包含 `<script>alert(1)</script>` 时不会执行脚本。
  - 核心按钮 Loading 锁生效，重复点击不会产生明显重复请求或重复状态流转。
- 记录浏览器 Network、Console 与页面截图证据。

### 4. 配置与依赖安全检查

- 检查 `.env`、JWT secret、数据库密码、Grafana 默认账号、Mock 配置是否适合验收与演示环境。
- 执行依赖安全扫描并记录结果：

```powershell
cd backend
govulncheck ./...

cd ../frontend
pnpm audit
```

- 如果本机未安装 `govulncheck`，先执行：

```powershell
go install golang.org/x/vuln/cmd/govulncheck@latest
```

### 5. 安全测试报告沉淀

- 新增 `docs/SECURITY_TEST.md`，记录：
  - 测试日期、测试人、测试环境。
  - 测试范围与非目标。
  - 自动化测试命令与结果。
  - 手工测试用例、预期结果、实际结果。
  - 发现的问题、风险等级、修复状态。
  - 剩余风险与验收说明。

### 6. 模糊测试补充

- 安全测试负责人需要补充模糊测试，用于验证接口在异常输入下的鲁棒性与安全边界。
- 模糊测试优先覆盖核心 API：
  - `/api/v1/auth/login`
  - `/api/v1/auth/register`
  - `/api/v1/activities`
  - `/api/v1/activities/:id`
  - `/api/v1/enrollments`
  - `/api/v1/enrollments/:id/status`
  - `/api/v1/orders/:id`
  - `/api/v1/orders/:id/pay`
  - `/api/v1/notifications/:id/read`
  - `/api/v1/recommendations`
- 输入扰动至少覆盖：
  - 空值、缺字段、多余字段。
  - 超长字符串、特殊字符、Unicode、控制字符。
  - SQL 注入式字符串。
  - XSS payload。
  - 负数、极大数字、浮点数、非数字 ID。
  - 错误 `Content-Type`。
  - 畸形 JSON。
  - 空 token、错误 token、超长 Authorization header。
- 验收口径：
  - 服务不 panic、不崩溃。
  - 不出现 5xx 风暴；单个异常请求若触发 5xx，必须记录并分析。
  - 不绕过认证、角色权限或资源归属校验。
  - 响应体不泄露 SQL、堆栈、密钥、DSN 或内部路径。
  - 数据库与 Redis 不因异常输入产生脏数据。
- 模糊测试结果需要写入 `docs/SECURITY_TEST.md`，至少包含测试范围、样例 payload、异常响应统计、发现问题与处理结论。

### 7. 实机运行环境与核心功能演示

- 两名实机演示负责人在正式演示前需要熟悉以下内容：
  - 项目启动顺序：Docker 依赖、后端、Seed、前端。
  - 核心账号：C 端用户、商户账号及其用途。
  - 核心业务闭环：商户发布活动、用户报名、订单生成、支付或取消、通知触达。
  - 关键服务关系：MySQL、Redis、Kafka、Prometheus、Grafana 在演示中的作用。
  - 常见故障处理：端口占用、Docker 未启动、后端健康检查失败、前端 Mock 未关闭、Kafka/Redis 异常、Grafana 无数据。
- 在验收使用的真实机器上实际拉起系统：
  - Docker 服务：MySQL、Redis、Kafka、Prometheus、Grafana。
  - 后端服务：`go run ./cmd/server`。
  - Seed 数据：`go run ./scripts/seed`。
  - 前端服务：`pnpm dev`，并确认 `VITE_USE_MOCK=false`。
- 固定并现场使用演示账号与演示数据：
  - C 端用户账号。
  - 商户账号。
  - 至少 1 个可报名活动。
  - 至少 1 条订单与通知可展示数据。
- 实际演示系统核心功能：
  - 后端健康检查返回 200。
  - 首页、活动列表、活动详情浏览。
  - 商户登录、创建活动、发布活动。
  - C 端用户登录、浏览活动、发起报名。
  - 报名进入排队或成功状态后，展示订单生成、支付或取消路径。
  - 展示通知铃铛、通知列表、订单列表、订单详情。
  - 展示推荐列表或热门活动模块。
  - 展示 Grafana Dashboard 与 Prometheus 指标。

### 8. 抗压能力展示

- 在真实运行环境中展示系统抗压能力，优先复用现有 JMeter 脚本：

```powershell
cd backend/tests/jmeter
bash run-jmeter-report.sh
```

- 抗压展示至少说明并验证：
  - 压测接口：`POST /api/v1/enrollments`。
  - 并发规模：按当前机器能力选择 1000 / 3000 / 5000 或已有 Sprint4 基线。
  - 成功判定：进入排队、业务性售罄、非预期失败的区分口径。
  - 关键指标：总请求数、错误率、P90/P95/P99、吞吐量。
  - 数据一致性：压测后库存不为负、不超卖、无重复成功报名。
  - 监控展示：Grafana 中 HTTP 吞吐、时延、Worker 成功/失败指标可观察。
- 如果现场机器不适合实时执行大规模压测，需要至少展示：
  - 已完成的 Sprint4 压测报告。
  - JMeter 脚本与参数。
  - Grafana/Prometheus 指标或历史截图。
  - 小规模现场压测结果，用于证明脚本和链路可运行。

### 9. 实机演示脚本与排练

- 输出一份可照着执行的实机演示脚本，覆盖：
  - Docker 依赖服务状态展示。
  - 后端健康检查与 Prometheus metrics 展示。
  - 商户端核心功能演示。
  - C 端核心功能演示。
  - 订单、通知、推荐模块展示。
  - JMeter 抗压能力展示。
  - Grafana 监控面板展示。
- 至少完成 1 次完整彩排，记录：
  - 彩排日期。
  - 使用机器与环境。
  - 操作路径。
  - 功能演示是否全部跑通。
  - 抗压展示是否能给出明确指标。
  - 是否出现卡顿、白屏、接口失败、控制台报错或压测异常。
  - 发现问题与处理结果。

## 四、验收标准 (DoD)

- 安全测试负责人完成并提交 `docs/SECURITY_TEST.md`。
- 后端安全测试覆盖认证、角色权限、水平越权与非法输入四类关键风险。
- 安全测试负责人完成核心 API 模糊测试，并在安全测试报告中记录样例 payload、异常响应统计与处理结论。
- `go test ./... -count=1` 通过，或报告中明确记录失败原因与处理结论。
- 前端手工安全走查完成，确认登录态、401 处理、Mock 关闭、XSS 展示风险无明显阻塞项。
- 依赖安全扫描完成，所有高危问题需给出修复或验收豁免说明。
- 两名实机演示负责人完成真实机器上的完整系统运行演示彩排，并确认核心功能、压测脚本、监控面板、演示账号、演示数据与服务状态可稳定复现。
- 两名实机演示负责人需要能独立说明并处理常见演示问题，至少包括服务启动顺序、账号用途、核心业务闭环、压测入口、监控面板含义与基础故障排查。
- 抗压能力展示必须给出明确指标，至少包含并发规模、总请求数、错误率、响应时间分位数与库存一致性结论。
- 若发现高危问题，必须完成修复验证后 Sprint 5 才可关闭；中低风险问题可记录为验收剩余风险。

## 六、协作说明

- 安全测试负责人负责发现、复现、记录与回归验证。
- 两名实机演示负责人负责保证系统在验收机器上可实际启动、可实际操作、可演示核心功能、可展示抗压能力。
- 两名实机演示负责人不负责验收材料、答辩材料、最终文档汇总或正式讲解稿准备。
- 后端或前端成员仅在安全测试发现缺陷后按模块协助修复。
- 本 Sprint 不扩大业务范围，不新增非必要功能。
- 涉及 API 契约、权限规则或状态机变更时，需同步更新 `docs/SYSTEM_DESIGN.md`；若影响范围较大，应补充 OpenSpec change。
