## 一、项目整体成果概览

UAAD 是一个面向活动发布、活动报名、订单支付、通知触达、个性化推荐与并发抢票场景的全栈系统。经过多轮 Sprint 迭代后，项目已经从基础需求分析、架构设计、前后端功能开发，推进到并发能力验证、安全测试、系统监控、实机演示和最终验收准备阶段。

最终系统形成了较完整的业务闭环：

1. 商户创建活动，活动可经历草稿、预热、发布、下架等状态。
2. 用户浏览活动列表、活动详情、热门活动和推荐活动。
3. 用户对已发布活动发起报名，系统通过 Redis 库存扣减和 Kafka 异步处理保证高并发下的库存一致性。
4. 报名成功后生成订单，用户可以支付订单或在允许状态下取消报名。
5. 系统通过通知模块向用户反馈报名成功、报名失败、订单过期、活动提醒等事件。
6. 推荐模块根据热度、用户行为和协同过滤结果生成活动推荐。
7. Prometheus 与 Grafana 提供系统运行指标观测，JMeter 与 Go benchmark 支持压力测试和抗压能力展示。
8. Sprint 5 补充安全测试、模糊测试和实机演示任务，为最终验收提供风险闭环。

## 二、已实现的功能

### 2.1 用户与认证模块

已实现功能：

- 用户注册。
- 用户登录。
- JWT 鉴权。
- 角色区分：普通用户与商户。
- 前端登录态保存、请求拦截、401 处理和退出登录。
- 后端受保护接口统一鉴权。
- 商户接口通过角色权限控制限制普通用户访问。

运行效果：

- 未登录访问受保护接口返回 401。
- 普通用户访问商户创建活动、发布活动、预热活动等接口返回 403。
- token 过期、格式错误、签名错误时会被拒绝。

### 2.2 活动管理模块

已实现功能：

- 活动创建。
- 活动列表查询。
- 活动详情查询。
- 商户活动列表。
- 活动编辑。
- 活动发布。
- 活动预热。
- 活动下架与过期活动处理。
- 活动分类、地点、价格、容量、开始时间、结束时间等基础字段管理。

活动状态：

- `DRAFT`：草稿状态。
- `PREHEAT`：预热状态，可用于展示或营销预热，但尚未正式开放报名。
- `PUBLISHED`：发布状态，用户可以报名。
- `OFFLINE`：下架或过期状态，不再作为可报名活动展示。

运行效果：

- 商户可以先创建草稿活动，再将活动设为预热。
- 预热活动可以继续发布为正式活动。
- 发布时系统会将库存写入 Redis，形成抢票链路所需的库存缓存。
- 过期活动可被自动下架逻辑处理。

### 2.3 报名与抢票模块

已实现功能：

- 用户报名活动。
- Redis 原子扣减库存。
- 重复报名幂等控制。
- 库存不足处理。
- Kafka 异步消息投递。
- Worker 消费报名消息并落库。
- 报名状态查询。
- 报名取消。
- 报名成功后生成订单。
- 报名失败后回滚或记录失败原因。

运行效果：

- 高并发报名下不会出现库存为负或明显超卖。
- 同一用户重复报名同一活动会被识别，避免重复扣减。
- Kafka 写入已调整为同步写入，降低“接口返回排队成功但消息未进入 Kafka”导致报名长期卡在 `QUEUING` 的风险。
- Go 并发测试、JMeter 压测和 Grafana 监控可以共同验证该链路。

### 2.4 订单模块

已实现功能：

- 报名成功后生成订单。
- 订单列表查询。
- 订单详情查询。
- 订单支付。
- 订单状态流转。
- 订单过期扫描。
- 订单过期后通知用户并回滚库存。

运行效果：

- 用户只能查看和支付自己的订单。
- 支付成功后订单状态更新。
- 过期订单会被后台逻辑处理，避免长期占用库存。

### 2.5 通知模块

已实现功能：

- 报名成功通知。
- 报名失败通知。
- 订单过期通知。
- 活动提醒通知。
- 通知列表查询。
- 未读通知数量查询。
- 标记通知已读。

运行效果：

- 用户在报名、支付、过期等关键流程后能够收到业务反馈。
- 用户不能读取或修改其他用户的通知。
- 未读数量能支持前端角标或通知入口展示。

### 2.6 推荐模块

已实现功能：

- 匿名用户热门推荐。
- 登录用户冷启动推荐。
- 用户行为足够时的协同过滤推荐。
- 热门活动接口。
- 用户行为上报。
- 活动热度分数计算。
- 推荐结果缓存与刷新。

推荐策略：

- 匿名用户或无行为用户：使用热门排序。
- 行为数量较少的用户：使用偏好分类补足热门活动。
- 行为数量较多的用户：使用基于物品的协同过滤，并结合热门和新鲜活动兜底。

运行效果：

- 首页或推荐页可以展示热门活动与个性化推荐活动。
- 浏览行为会影响活动热度和后续推荐结果。

### 2.7 前端界面与交互

已实现功能：

- 首页与活动列表。
- 活动详情页。
- 登录与注册。
- 商户活动管理。
- 活动创建、编辑、发布、预热入口。

## 三、掌握的关键技术

### 3.1 后端技术

项目后端主要使用 Go 技术栈：

- Go 语言。
- Gin Web 框架。
- GORM ORM。
- MySQL 数据库。
- Redis 缓存与库存扣减。

关键学习点：

- 如何设计 RESTful API。
- 如何进行用户认证与角色鉴权。
- 如何使用 Redis Lua 脚本保证库存扣减原子性。
- 如何使用 Kafka 解耦高并发报名请求和后端落库处理。
- 如何通过事务、幂等和状态机保证业务一致性。
- 如何通过 Go 测试覆盖 service、handler、worker 和集成链路。

### 3.2 前端技术

项目前端主要使用 React 技术栈：

- React。
- TypeScript。
- Vite。
- React Router。
- Axios。
- Vitest。
- Tailwind CSS。
- ECharts。
- i18next。

关键学习点：

- 如何组织前端路由和页面结构。
- 如何封装 API 请求与统一错误处理。
- 如何处理登录态、token 和 401 跳转。
- 如何将商户端和用户端操作入口进行角色区分。
- 如何通过 Vitest 测试 API 封装和基础页面能力。
- 如何在验收阶段关闭 Mock，使用真实后端进行联调。

### 3.3 数据与中间件技术

使用到的中间件：

- MySQL：持久化用户、活动、报名、订单、通知、行为、推荐分数等核心数据。
- Redis：缓存活动库存、报名用户集合、库存预热标记等。
- Kafka：承接报名消息，削峰填谷，降低高并发请求对数据库的直接压力。
- Prometheus：采集后端指标。
- Grafana：展示监控面板。

关键学习点：

- 数据库适合持久化事实数据。
- Redis 适合承接高频读写和库存原子扣减。
- Kafka 适合将高峰请求异步化，减少同步链路压力。
- 监控系统需要提前埋点，否则问题发生时很难解释系统行为。

## 四、开发流程与团队协作

### 4.1 整体开发流程

项目采用按 Sprint 推进的迭代方式：

1. 需求分析阶段：整理 SRS，明确用户角色、功能范围和非功能要求。
2. 架构设计阶段：设计前后端架构、数据库结构、API 契约和核心数据流。
3. 基础功能阶段：完成用户、活动、报名、订单、通知等基础模块。
4. 并发能力阶段：引入 Redis、Kafka、库存扣减、异步 worker 和压测脚本。
5. 推荐与体验阶段：补充推荐模块、前端页面和业务闭环体验。
6. 验收准备阶段：补充安全测试、模糊测试、监控、实机演示和压测展示。
7. 总结反思阶段：整理最终成果、缺陷复盘、课程建议、个人心得和配套材料。

### 4.5 沟通机制

项目协作主要依赖以下机制：

- 使用 Sprint 文档明确阶段目标、任务范围和负责人。
- 使用 `docs/README.md` 作为文档入口，减少材料散落。
- 使用 `docs/SYSTEM_DESIGN.md` 记录稳定架构和 API 契约。
- 使用 OpenSpec 管理非琐碎变更，避免需求口头漂移。
- 使用测试命令和运行指南作为联调依据，而不是只依赖口头说明。
- 对复杂问题先复现，再定位，再修复，再运行测试验证。

### 4.6 协作中暴露的问题

主要问题：

- 文档与代码同步有时滞后，导致后续成员需要重新确认真实实现。
- 早期任务边界不够清晰，容易出现某些模块无人持续跟进。
- 部分功能先实现后补测试，导致验收前补测压力较大。
- 前端、后端、测试和演示之间的运行环境要求没有一开始就统一，后期出现端口、数据库、pnpm workspace、seed 数据等问题。
- 对 Kafka、Redis、Prometheus、Grafana 等中间件的依赖关系，早期没有完全用图或脚本固化，导致新人理解成本较高。
- 智能体生成内容需要人工审核，若直接采用，可能出现路径不一致、命令不适配或材料过度泛化。

改进方向：

- 每个 Sprint 结束时同步更新代码、测试、运行说明和已知风险。
- 每个功能合并前补齐最小测试，不把所有测试压力留到验收前。
- 关键链路固定一份“从零拉起系统”的脚本和检查清单。
- 对智能体输出建立人工复核机制，尤其是状态机、权限、并发和安全相关内容。
- 让每个成员至少掌握系统启动、核心业务闭环和常见故障处理，而不是只了解自己负责的文件。

## 六、缺陷与反思

### 6.1 已发现并处理的问题

| 问题 | 表现 | 处理方式 | 反思 |
|---|---|---|---|
| 数据库连接失败 | 后端启动时报 MySQL 连接拒绝或 root 权限错误 | 检查 Docker MySQL、端口映射、本地 3306 占用和 `.env` 配置 | 运行依赖应尽早脚本化和文档化 |
| 前端 pnpm 启动失败 | `packages field missing or empty` | 修复 `frontend/pnpm-workspace.yaml` | 工具链配置也属于验收风险 |
| 报名可能长期卡在 `QUEUING` | Kafka 异步写入可能导致接口先返回成功但消息未实际投递 | 将 Kafka Writer 调整为同步写入并设置超时 | 异步系统不能只看接口返回，还要关注消息可靠性 |
| 活动预热状态原先只有概念 | `PREHEAT` 没有真实设置入口 | 新增后端预热接口和前端商户操作 | 状态机中的每个状态都应有明确进入路径 |
| 安全测试初期覆盖不足 | 验收前才集中补安全与模糊测试 | Sprint 5 补充安全授权测试和模糊测试 | 安全测试应更早进入迭代节奏 |
| 压测链路理解成本高 | JMeter、Grafana、Prometheus、Kafka 指标较多 | 在文档中固定压测命令、指标解释和演示路径 | 抗压能力展示要面向评审可理解，而不只是能跑脚本 |

### 6.2 使用智能体过程中的心得体会

正向作用：

- 智能体能快速读取仓库、定位文件、总结功能和整理文档。
- 对错误日志的解释和排查路径很有帮助，尤其是 Docker、MySQL、Kafka、pnpm 等问题。
- 对跨模块变更有帮助，例如活动预热同时涉及后端 service、handler、routes、前端 API、页面操作和文档。
- 对测试补充有帮助，可以根据验收口径生成安全测试、模糊测试、集成测试和运行命令。
- 对最终材料整理很有帮助，可以把分散的文档、测试、报告和运行说明汇总为提交结构。

暴露的问题：

- 智能体可能根据一般经验给出建议，但项目真实代码不一定一致，因此必须要求其先读仓库。
- 智能体容易生成“看起来完整”的文档，但其中的人名、日期、测试结果、截图证据必须由团队核实。
- 智能体不能代替团队理解业务状态机和系统边界，尤其是报名、库存、订单、支付、取消这些关键链路。
- 智能体生成的代码仍需要测试验证，不能只靠文字解释判断正确。

改进经验：

- 提问时给出当前错误日志、所在目录、运行命令和期望结果。
- 让智能体先查文件再回答，减少幻觉。
- 对每次变更要求给出验证命令。
- 对验收材料要求区分“已实现”“已测试”“已记录风险”和“未来建议”。
- 对个人心得、课程反馈等主观材料，由成员自己补充真实经历，智能体只负责结构化。

### 6.3 团队协作反思

本项目最大收获之一是理解到：系统开发不是把功能分别写完就结束，而是要让功能、测试、运行、监控、文档和演示共同闭环。

反思点：

- 如果没有统一运行指南，系统即使能在某个人电脑上跑，也不代表能在验收现场稳定运行。
- 如果没有测试和监控，高并发报名这类功能很难证明“真的可靠”。
- 如果没有明确分工，后期容易出现大家都以为别人会补文档、补测试、补演示脚本的情况。
- 如果文档没有持续维护，最终总结阶段会花大量时间重新追溯项目事实。
- 如果每个成员只了解自己负责的局部，实机演示时很难应对临时问题。

后续类似项目应在一开始就建立：

- 固定任务看板。
- 每周可运行版本。
- 每个 Sprint 的 DoD。
- 最小测试要求。
- 文档入口和材料命名规范。
- 演示脚本和故障排查清单。

## 七、课程建议

### 7.1 对课程内容的建议

建议增加以下内容：

- 增加“从需求到可运行系统”的完整样例，展示如何从 SRS、架构设计、API、数据库、测试、部署一步步推进。
- 增加 AI Agent 辅助开发的边界说明，特别是如何避免智能体幻觉、如何审查智能体生成代码、如何记录智能体交互过程。
- 增加中间件实践讲解，例如 Redis、Kafka、Prometheus、Grafana 在一个小型系统中分别解决什么问题。
- 增加安全测试基础内容，包括认证、鉴权、越权、XSS、SQL 注入、模糊测试和依赖漏洞扫描。
- 增加压测结果解释方法，不只教如何运行工具，还要教如何看 P95、P99、错误率、吞吐量和资源瓶颈。

### 7.2 对组织形式的建议

建议课程组织上：

- 每个阶段明确必交材料，例如需求文档、系统设计、测试报告、运行指南、复盘报告。
- 每个 Sprint 增加一次短检查，确认系统是否能从零拉起。
- 让团队尽早确定角色分工，但允许根据实际进展调整。
- 要求每个成员至少完成一次代码讲解或演示讲解，避免少数成员承担全部理解成本。
- 验收前增加一次模拟验收，提前暴露环境、脚本、数据和讲解问题。

### 7.3 对教学方法的建议

建议教学方法上：

- 多给“可复现”的评价标准，例如必须提供运行命令、测试命令、截图或日志。
- 对 AI 辅助开发过程设置记录要求，例如保留关键对话、关键 prompt、智能体生成内容和人工修改说明。
- 鼓励学生使用真实工程工具，如 Docker、CI、测试框架、监控系统，而不仅是完成页面和接口。
- 对复杂系统设计引入阶段性评审，帮助学生尽早发现架构过度设计或实现不足的问题。
- 对最终报告提供模板，但保留团队自由发挥空间。


关注指标：

- 总请求数。
- 成功率与错误率。
- P90 / P95 / P99 响应时间。
- 每秒吞吐量。
- 库存一致性。
- Kafka lag。
- Worker success / failure。

实际表现总结：

- 系统具备通过 Redis 原子扣减和 Kafka 异步处理承接并发报名的能力。
- 压测过程中需要区分“业务性失败”和“系统性失败”。例如库存售罄属于业务预期，不应简单计为系统错误。
- Grafana 能帮助解释压测期间请求量、延迟、错误和 Worker 处理情况。

优化方向：

- 增加更多压测场景，例如活动列表、推荐接口、订单支付、通知查询。
- 固定不同机器规格下的压测基线。
- 增加压测前自动清理数据和压测后自动一致性校验。
- 将 JMeter 报告和 Grafana 截图按日期归档。

### 8.2 微服务架构

当前实践情况：

- 当前系统仍以单体后端为主，没有完全拆分为多个独立微服务。
- 但系统内部已经具备一定的服务化边界，包括活动、报名、订单、通知、推荐、行为、Worker 等模块。
- Kafka Worker 已经体现了异步服务化思路：报名请求与后端落库处理被消息队列解耦。

潜在拆分策略：

| 服务 | 职责 | 数据/通信 |
|---|---|---|
| 用户认证服务 | 注册、登录、JWT、用户资料 | MySQL、HTTP |
| 活动服务 | 活动创建、发布、预热、下架、查询 | MySQL、Redis |
| 报名服务 | 报名请求、库存扣减、幂等控制 | Redis、Kafka、MySQL |
| 订单服务 | 订单生成、支付、过期扫描 | MySQL、消息事件 |
| 通知服务 | 通知创建、未读数、已读状态 | MySQL、消息事件 |
| 推荐服务 | 行为收集、热度计算、推荐生成 | MySQL、缓存 |
| 监控服务 | 指标采集、日志聚合、告警 | Prometheus、Grafana |

推荐技术路径：

- 第一阶段：保持模块化单体，明确 service、repository、handler 边界。
- 第二阶段：将 Worker 独立进程化，报名 API 与消费处理分离部署。
- 第三阶段：将推荐服务、通知服务拆出，使用 HTTP 或 gRPC 通信。
- 第四阶段：引入服务发现、网关、链路追踪和集中配置。

收益：

- 模块独立部署。
- 不同服务可按负载独立扩容。
- 推荐、通知、订单等模块可独立演进。
- 高并发报名链路可以单独优化。

风险：

- 分布式事务复杂度增加。
- 部署和监控成本提升。
- 本课程项目规模下完全微服务化可能超过实际需要。

### 8.3 系统监控

已部署方案：

- Prometheus 采集后端指标。
- Grafana 展示系统监控 Dashboard。
- 后端暴露 `/metrics`。
- Docker Compose 管理 Prometheus 与 Grafana。

相关文件：

- `infra/grafana/provisioning/dashboards/uaad-sprint3.json`
- `docs/Prometheus_And_Grafna.md`
- `docker-compose.yaml`

主要面板：

- HTTP 吞吐。
- HTTP 状态码分布。
- 总请求量。
- 5xx 错误数。
- HTTP P50 / P95 / P99。
- 按 Path 的 P95。
- Worker 处理吞吐。
- Worker 耗时 P95。
- Kafka 消费滞后。
- Worker 处理消息数。

实际成效：

- 在压测和演示中可以直观看到请求量上升。
- 可以通过状态码分布判断是否出现大量非预期错误。
- 可以通过 Worker 指标判断 Kafka 消息是否被正常消费。
- 可以通过 Kafka lag 判断异步处理是否堆积。

改进建议：

- 增加 MySQL、Redis、Kafka 自身指标采集。
- 增加日志聚合，例如 Loki 或 ELK。
- 增加告警规则，例如 5xx 过高、P95 过高、Kafka lag 过高。
- 区分业务错误和系统错误的监控口径。
- 生产环境下对 `/metrics` 做网络隔离或认证保护。

### 8.4 弹性设计

当前已采取的措施：

- Redis 原子扣减库存，降低数据库直接承压。
- Kafka 异步削峰，避免报名请求直接把数据库打满。
- Worker 异步处理报名消息。
- 报名幂等控制，降低重复请求风险。
- 订单过期扫描和库存回滚，避免资源长期占用。
- Kafka 同步写入和超时设置，降低消息丢失导致状态卡住的风险。
- 健康检查和监控面板辅助发现运行问题。

已有收益：

- 高并发报名链路具备一定抗压能力。
- 库存扣减更加稳定。
- 请求处理和落库处理解耦。
- 异步 worker 可作为后续独立扩容点。

不足：

- 暂未引入断路器。
- 暂未系统性实现重试策略和死信队列。
- 暂未实现服务自动扩容。
- 暂未实现负载均衡部署。
- 暂未实现统一降级策略。

未来可引入的弹性策略：

- 对 Kafka 消费失败引入重试和死信队列。
- 对外部依赖引入超时、重试和熔断。
- 对推荐等非核心接口提供降级结果，例如返回热门活动。
- 将 Worker 独立部署并允许多实例水平扩展。
- 使用 Nginx 或 API Gateway 做负载均衡。
- 使用容器编排平台实现自动恢复。

### 8.5 安全性保障

现有安全措施：

- JWT 身份认证。
- bcrypt 密码哈希。
- 角色权限控制。
- 用户私有资源归属校验。
- GORM 参数化查询降低 SQL 注入风险。
- 前端未使用 `dangerouslySetInnerHTML`，降低 XSS 执行风险。
- CORS 配置支持开发和生产区分。
- 安全授权测试覆盖 401、403、越权、非法输入、信息泄露和 CORS。
- 模糊测试覆盖核心 API 的异常输入。
- 依赖安全扫描覆盖 Go 和前端依赖。

相关文件：

- `backend/tests/security_authz_test.go`
- `backend/tests/security_fuzz_test.go`
- `docs/SECURITY_TEST.md`

安全测试运行方式：

```powershell
cd backend
go test -v -tags=bgroup -run '^TestSec|^TestFuzz' -count=1 ./tests/
```

已知不足：

- JWT secret 存在开发默认值，生产环境必须覆盖。
- JWT 存储在 localStorage，仍有 XSS 场景下被窃取的风险。
- 登录接口限流仍需进一步加强。
- `/metrics` 在演示环境中公开，生产环境需隔离。
- 部分异常输入可能返回 500，说明输入校验还可以更前置、更精细。
- 服务端 token 撤销机制尚未实现。

后续加固方向：

- 启动时强制检查生产环境 JWT secret。
- 增加登录限流和验证码策略。
- 对注册、登录、活动创建等接口加强字段格式和长度校验。
- 使用 HttpOnly Cookie 或更严格的 token 存储策略。
- 增加 CSRF 防护设计。
- 增加敏感操作审计日志。
- 对 Prometheus metrics 做认证或内网隔离。
- 引入 OWASP ZAP 等动态扫描工具。
